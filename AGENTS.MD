# Coding Agent Playbook (Spec Kit)

This repository uses GitHub Spec Kit feature packages under `specs/`.
Agents must follow the Spec Kit workflow for both specification changes and implementation work.

Codex and other agents discover instructions by walking from the current directory up to the repo root and reading `AGENTS.md` files (if subdirectory overrides exist, the most specific file wins).

---

## 0) Prime directive

1. Keep context lean. Read only the files needed for the current feature and phase.
2. Prefer small, reviewable diffs. Edit existing artifacts instead of rewriting unless structure is wrong.
3. Keep code and spec artifacts synchronized. Do not let implementation drift from `spec.md`, `plan.md`, or `tasks.md`.
4. Do not invent product requirements. If something critical is missing, state assumptions explicitly or ask focused questions.

---

## 1) Repository map (actual)

- `specs/<NNN-feature-slug>/`
  - `spec.md` (requirements and user stories)
  - `plan.md` (technical plan and architecture choices)
  - `research.md` (decisions and tradeoffs)
  - `data-model.md`
  - `contracts/`
  - `quickstart.md`
  - `tasks.md` (execution checklist)
- `.specify/`
  - `memory/constitution.md` (governing principles)
  - `templates/` (Spec Kit templates)
  - `scripts/bash/` and `scripts/powershell/` (local workflow scripts)
- `.codex/prompts/speckit.*.md` and `.claude/commands/speckit.*.md`
  - Agent command packs generated by Spec Kit.
- `docker/speckit/`
  - Containerized bootstrap and `specify` command usage.

If adding a new feature package, also update `specs/README.md`.

---

## 2) Feature and branch conventions

1. Feature branches must be named `NNN-short-name` (example: `003-branching-control-flow`).
2. Feature folder must match branch name: `specs/NNN-short-name/`.
3. Scripts resolve active feature from branch name or `SPECIFY_FEATURE` env var.
4. Do not mix work for multiple feature packages in a single change unless explicitly requested.

---

## 3) Required workflow

### A) Create or refine spec

1. `/speckit.specify` to create/update `spec.md`.
2. `/speckit.plan` to produce `plan.md`, `research.md`, `data-model.md`, `contracts/`, `quickstart.md`.
3. `/speckit.tasks` to generate dependency-ordered `tasks.md`.
4. Optional but recommended: `/speckit.analyze` before implementation.

### B) Implement

1. Run prerequisites check from repo root:
   - Bash: `bash .specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks`
   - PowerShell: `powershell -ExecutionPolicy Bypass -File .specify/scripts/powershell/check-prerequisites.ps1 -Json -RequireTasks -IncludeTasks`
2. Read the current feature's `tasks.md`, `plan.md`, `spec.md`, then supporting docs (`data-model.md`, `contracts/`, `research.md`, `quickstart.md`) as needed.
3. Execute work in task order and phase order unless dependencies clearly allow parallel work.
4. Mark completed tasks as `- [x]` in `tasks.md`.
5. Re-run tests and quickstart scenarios relevant to completed tasks.

### C) Convert tasks to issues (optional)

- Use `/speckit.taskstoissues` only when remote is a GitHub repository and issue creation is requested.

---

## 4) Constitution and quality gates

The constitution at `.specify/memory/constitution.md` is authoritative. In particular:

1. Source-first contract: JSON/YAML source is the canonical exchange.
2. No runtime network calls from editor core.
3. Editor owns validation and diagnostics behavior.
4. Accessibility baseline is mandatory for core flows.
5. Compatibility/versioning and extensibility boundaries must remain explicit.

If implementation requires violating a constitution principle, stop and request an explicit constitution or scope change instead of silently proceeding.

---

## 5) Implementation discipline for agents

1. Treat `tasks.md` as the execution ledger. Keep it current.
2. If code changes require design changes, update the feature package docs in the same change:
   - requirement change -> `spec.md`
   - architectural/tech choice change -> `plan.md` and `research.md`
   - API/data contract change -> `contracts/` and/or `data-model.md`
   - validation scenario change -> `quickstart.md`
3. Keep changes scoped to the active feature package plus necessary code files.
4. Prefer linking to existing docs over duplicating long prose in new files.

---

## 6) Status checks agents should run

From repo root:

1. Feature readiness:
   - Bash: `bash .specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks`
   - PowerShell: `powershell -ExecutionPolicy Bypass -File .specify/scripts/powershell/check-prerequisites.ps1 -Json -RequireTasks -IncludeTasks`
2. Spec lifecycle status:
   - read `**Status**:` in `specs/<feature>/spec.md`
3. Task progress:
   - count checked vs unchecked items in `specs/<feature>/tasks.md`

Use these checks in progress updates and final summaries.

---

## 7) Environment notes

1. If PowerShell blocks local scripts, run Spec Kit scripts with `-ExecutionPolicy Bypass`.
2. If git reports "dubious ownership", configure safe directory for this repo before branch-based script operations.
3. Docker is optional; if unavailable, use local `.specify/scripts/bash/*.sh` or `.specify/scripts/powershell/*.ps1` scripts directly.

---

## 8) Subdirectory overrides

Subdirectories may define stricter instructions in their own `AGENTS.md`.
Always apply the most specific `AGENTS.md` for the file(s) being edited.
